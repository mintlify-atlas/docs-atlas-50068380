Dolibarr is a PHP web application built without a heavy MVC framework. Every URL maps directly to a PHP script inside `htdocs/`, which acts as the web root. The codebase is organized by feature rather than by layer: each business domain (invoices, orders, third parties) lives in its own directory alongside its classes, templates, and SQL. The result is a system where you can follow a feature from its UI page down to its database query by navigating a single directory tree.

## Directory layout

```text
dolibarr/
└── htdocs/                        # Web root — served directly by the web server
    ├── core/                      # Core framework code
    │   ├── lib/                   # Global utility functions (functions.lib.php, …)
    │   ├── class/                 # Framework classes (CommonObject, HookManager, …)
    │   ├── modules/               # Module descriptor base class (DolibarrModules)
    │   ├── triggers/              # Built-in trigger implementations
    │   └── ajax/                  # AJAX endpoint handlers
    ├── api/                       # REST API bootstrap (Luracast Restler)
    │   └── class/                 # DolibarrApi base class + auto-loaded API classes
    ├── conf/                      # Configuration (conf.php generated by install wizard)
    ├── theme/                     # UI themes (eldy, md)
    ├── langs/                     # Translations — one dir per locale (en_US/, fr_FR/, …)
    ├── custom/                    # Third-party and custom modules (not shipped with core)
    ├── societe/                   # Third-parties (customers, suppliers) module
    ├── commande/                  # Sales orders module
    ├── compta/facture/            # Customer invoices module
    └── … (one dir per feature)
```

## Core architectural concepts

<Columns cols={2}>
  <Card title="No heavy framework" icon="layers">
    Dolibarr does not use Symfony, Laravel, or any MVC container. Routing is filesystem-based: the URL path matches the PHP file path under `htdocs/`. Global objects (`$db`, `$conf`, `$user`, `$langs`) are initialized in a shared `main.inc.php` and made available via `global`.
  </Card>
  <Card title="CommonObject base class" icon="cube">
    Every business entity (invoice, order, product, third party) extends `CommonObject` (`htdocs/core/class/commonobject.class.php`). It provides shared behavior: `create()`, `fetch()`, `update()`, `delete()`, extra-fields support, linked-object management, and trigger firing via the `CommonTrigger` trait.
  </Card>
  <Card title="DolibarrModules descriptor" icon="puzzle">
    Each module is described by a class that extends `DolibarrModules` (`htdocs/core/modules/DolibarrModules.class.php`). The descriptor declares menus, permissions, triggers, hooks, dictionaries, and SQL to run on activation. Dolibarr discovers descriptors by scanning for files matching `mod*.class.php`.
  </Card>
  <Card title="Database abstraction (DoliDB)" icon="database">
    `DoliDB` wraps PDO/MySQLi and provides a unified query interface (`query()`, `fetch_object()`, `escape()`, etc.). It supports MariaDB, MySQL, and PostgreSQL. All table names are prefixed with `llx_` by convention. The database object is always available as `$db`.
  </Card>
  <Card title="REST API (Luracast Restler)" icon="globe">
    The `htdocs/api/` directory bootstraps Luracast Restler 3. API classes extend `DolibarrApi` and are auto-discovered. Routes and parameter types are inferred from PHPDoc annotations (`@param`, `@return`, `@url`). Authentication uses Dolibarr API keys passed as `DOLAPIKEY` or via HTTP Basic.
  </Card>
  <Card title="Hook and trigger system" icon="zap">
    Hooks let modules inject output or modify queries at named points in core pages (UI-time, synchronous). Triggers fire after business events complete (e.g., `BILL_VALIDATE`) and are called by `CommonObject::call_trigger()`. Both mechanisms allow extending Dolibarr without editing core files.
  </Card>
</Columns>

## Database layer

Dolibarr's `DoliDB` class is the single point of database access across the application. It normalizes differences between MySQL/MariaDB and PostgreSQL and provides helpers for common patterns:

- `$db->query($sql)` — execute a raw SQL string; returns a result resource
- `$db->fetch_object($result)` — fetch a row as a PHP object
- `$db->escape($value)` — escape a value for safe SQL interpolation
- `$db->begin()` / `$db->commit()` / `$db->rollback()` — transaction control
- `$db->idate($timestamp)` — convert a Unix timestamp to the database date format

All table names use the `llx_` prefix (configurable in `conf.php`). Typed field definitions via `$fields` arrays (introduced in Dolibarr v14) allow `CommonObject` to auto-generate INSERT/UPDATE queries without manual SQL in each class.

## Template system

Dolibarr does not use a dedicated template engine such as Twig or Smarty. Pages are plain PHP files that mix logic and HTML. Reusable fragments are PHP files stored in `tpl/` subdirectories and included with `include`. Form helpers are available through the `Form` class (`htdocs/core/class/html.form.class.php`) and its domain-specific variants (e.g., `FormOrder`, `FormAccounting`).

## REST API

```php
// htdocs/api/index.php (simplified bootstrap)
use Luracast\Restler\Restler;

$r = new Restler($production_mode);
$r->addAPIClass('Invoices');   // auto-maps GET /invoices, POST /invoices, etc.
$r->addAPIClass('Orders');
// … additional classes discovered from enabled modules
$r->handle();
```

The `DolibarrApi` constructor (`htdocs/api/class/api.class.php`) instantiates Restler, configures the cache directory, and sets the base URL. Each API class registers its routes by declaring methods with PHPDoc `@url` annotations or by following Restler's RESTful naming convention. API modules place their class in `htdocs/{module}/class/api_{module}.class.php`.

## Multi-language support

Language strings are stored in INI-style `.lang` files under `htdocs/langs/{LANG_COUNTRY}/` (e.g., `htdocs/langs/en_US/main.lang`). The global `$langs` object (class `Translate`) loads these files on demand via `$langs->load('filename')` and translates keys with `$langs->trans('KeyName')`. Modules ship their own lang files in the same structure, loaded automatically when the module is enabled.

## Hook and trigger overview

| Mechanism | When it fires | Primary use |
|-----------|--------------|-------------|
| **Hook** | During page rendering or query building | Inject HTML, add form fields, filter SQL |
| **Trigger** | After a business event completes | Send notifications, sync external systems, write audit logs |

See [Hooks and triggers](/developer/hooks-triggers) for implementation details.
